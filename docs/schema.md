# Task Manager CLI - Data Schema and File Structure

## Purpose

This document defines the in-memory data schema and source file layout for the Task Manager CLI described in `docs/project-plan.md`.

## Technology and Constraints

- Runtime: Node.js 20+
- Language: JavaScript (ES modules)
- Dependencies: built-in Node.js modules only
- Persistence: in-memory only (no database, no file persistence)

## Data Schema

### Enumerations

#### TaskStatus
- `todo`
- `in-progress`
- `done`

#### TaskPriority
- `low`
- `medium`
- `high`

### Entity: Task

Represents a single task managed by the CLI.

| Field | Type | Required | Default | Rules |
|---|---|---|---|---|
| id | number | yes | auto-generated | Positive integer, unique, immutable after creation |
| title | string | yes | none | Trimmed, non-empty, max length 100 |
| description | string | no | '' | Trimmed, max length 500 |
| status | TaskStatus | yes | `todo` | Must be one of allowed status values |
| priority | TaskPriority | yes | `medium` | Must be one of allowed priority values |
| category | string | no | `general` | Trimmed, non-empty, max length 50 |
| createdAt | string (ISO 8601) | yes | generated at create time | Immutable after creation |
| updatedAt | string (ISO 8601) | yes | generated at create time | Must be updated on any task mutation |

Example:

```js
{
  id: 1,
  title: 'Write workshop notes',
  description: 'Summarize exercise outcomes',
  status: 'todo',
  priority: 'high',
  category: 'workshop',
  createdAt: '2026-03-31T10:15:00.000Z',
  updatedAt: '2026-03-31T10:15:00.000Z'
}
```

### Aggregate: TaskStore

In-memory state container for all tasks.

| Field | Type | Required | Description |
|---|---|---|---|
| tasks | Task[] | yes | Ordered collection of task records |
| nextId | number | yes | Next ID to assign; initialized to 1 |

Invariants:
- `nextId` is always greater than any existing task ID.
- IDs are never reused after deletion.
- `tasks` may be empty.

Example:

```js
{
  tasks: [],
  nextId: 1
}
```

## Validation Rules

### Task Property Validation Matrix

The following rules apply to each Task property during create and update operations.

| Property | Type | Create Rules | Update Rules | Failure Condition |
|---|---|---|---|---|
| id | number | System-generated from `nextId`; must be integer > 0 and unique | Not user-editable; immutable | Non-integer/<=0 ID input, duplicate ID, or attempt to patch `id` |
| title | string | Required; trim before validation; length 1-100 after trim | Optional in patch; if provided, trim and enforce length 1-100 | Missing on create, empty after trim, or >100 chars |
| description | string | Optional; defaults to ''; trim; max length 500 | Optional in patch; if provided, trim and enforce max length 500 | Non-string value or >500 chars |
| status | enum (`todo`/`in-progress`/`done`) | Optional; defaults to `todo`; normalize to lowercase; must match allowed set | Optional in patch; normalize to lowercase; must match allowed set | Value outside allowed status set |
| priority | enum (`low`/`medium`/`high`) | Optional; defaults to `medium`; normalize to lowercase; must match allowed set | Optional in patch; normalize to lowercase; must match allowed set | Value outside allowed priority set |
| category | string | Optional; defaults to `general`; trim; length 1-50 | Optional in patch; if provided, trim and enforce length 1-50 | Non-string value, empty after trim, or >50 chars |
| createdAt | ISO 8601 string | Set by system at creation via `new Date().toISOString()` | Not user-editable; immutable | Missing system timestamp at create or any patch attempt |
| updatedAt | ISO 8601 string | Set by system at creation; initially same as `createdAt` | Must be set by system on every successful mutation | Missing refresh after update/delete-adjacent mutation |

Normalization notes:
- Whitespace normalization is applied before length checks on string fields.
- Enum inputs are case-insensitive at command input boundary and stored in canonical lowercase.
- Unknown properties in update payloads should be rejected to prevent silent data drift.

### Create Task
- `title` is required.
- `status` defaults to `todo` if omitted.
- `priority` defaults to `medium` if omitted.
- `category` defaults to `general` if omitted.
- `createdAt` and `updatedAt` are generated by the system at creation.

### Update Task
- `id` must exist.
- At least one mutable field must be provided: `title`, `description`, `status`, `priority`, or `category`.
- `createdAt` cannot be changed.
- `updatedAt` must be refreshed after a successful update.

### Delete Task
- `id` must exist.
- Deletion removes exactly one task.

### Filter and Sort Inputs
- Filter by `status` accepts only TaskStatus values.
- Filter by `priority` accepts only TaskPriority values.
- Filter by `category` accepts non-empty string values.
- Sort key accepts only `priority` or `createdAt`.
- Priority sorting order: `high`, `medium`, `low`.
- Creation date sorting default: newest first.

## Error Handling Schema

### Error Types

- `ValidationError`
  - For invalid inputs (missing title, invalid enum values, malformed ID).
- `NotFoundError`
  - For operations on a non-existent task ID.
- `InvalidOperationError`
  - For unsupported command usage or invalid sort/filter combinations.

### Error Payload Shape

```js
{
  name: 'ValidationError',
  message: 'Priority must be one of: low, medium, high.',
  code: 'VALIDATION_ERROR'
}
```

Conventions:
- Throw `Error` objects (or subclasses), never plain strings.
- Use `console.error` for error logging.
- Non-zero process exit code for failed commands.

## File Structure

Proposed directory layout under `src/`:

```text
src/
  index.js
  constants.js
  types.js
  errors.js
  validators.js
  store.js
  filters.js
  commands.js
  formatters.js
```

### File Responsibilities

- `src/index.js`
  - CLI entry point.
  - Parses argv and dispatches commands.
  - Handles top-level error-to-exit-code mapping.

- `src/constants.js`
  - Shared constants such as allowed statuses and priorities.
  - Priority ranking map for sorting.

- `src/types.js`
  - JSDoc typedefs for `Task`, `TaskStore`, command input shapes.

- `src/errors.js`
  - Custom error classes: `ValidationError`, `NotFoundError`, `InvalidOperationError`.

- `src/validators.js`
  - Input normalization and validation functions.
  - ID parsing and enum checks.

- `src/store.js`
  - In-memory state and CRUD operations on `TaskStore`.
  - Timestamp and ID assignment logic.

- `src/filters.js`
  - Filtering by status/priority.
  - Sorting by priority/createdAt.

- `src/commands.js`
  - Command handlers that compose store, validator, and filter logic.

- `src/formatters.js`
  - CLI output formatting for lists, details, and error messages.

## Module Interfaces (Suggested)

```js
export function filterTasksByCategory(store, category);

// filters.js
export function filterTasks(tasks, { status, priority, category });
export function sortTasks(tasks, { sortBy, direction });

// validators.js
export function validateCreateInput(input);

export function validateTaskId(value);
```

- Validation functions
- Store CRUD functions
- Filter and sort behavior
- Command error mapping and exit behavior
